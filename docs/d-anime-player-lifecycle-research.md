# dアニメストア プレイヤーライフサイクル調査

調査日: 2026-07-23

## 調査境界

- 外部接続を使うのは、ログイン済みChromeから公式JavaScriptとマイページDOMを取得する工程だけとする。
- マイページDOMは、識別子、入力値、外部画像URL、イベント属性、インラインスタイルを除去したモックとして保存する。
- 動作研究は取得済みファイルだけを読み、Chromiumをオフラインに設定し、HTTP(S)ルーティングも遮断したローカルDOMで行う。
- 認証情報、Cookie、作品名、作品・話数の識別子、動画URLは研究成果物へ保存しない。

## 取得物

取得した公式プレイヤー資産は `test-fixtures/d-anime/official-assets-manifest.json` にファイル名、サイズ、SHA-256を記録した。実体はGit管理外の `.d-anime-sandbox/official-assets/` に置く。

ローカル再現で実行する `player.min-20260319174448.js` のSHA-256は次のとおり。

```text
99a1886d92e812089481e6546798df8758d03385872024787ee3aa085d1acd98
```

再現前に実体のSHA-256をマニフェストと照合し、不一致なら停止する。

## 公式プレイヤーの状態契約

`scripts/replay-d-anime-official-phases.mjs` は公式バンドルから次の実装を抽出して、そのままローカルDOMで実行する。

- `initializeSub1st`: プレイヤー生成後に `player.play()` を呼ぶ。
- `destroyPlayer`: 現プレイヤーを破棄し、破棄完了後に初期化へ戻る。
- `SkipUI`: チャプター移動、次話移動、再生終了を分岐する。
- `procEndedEvent`: 終端時に再生位置を保存し、動画を停止する。

再現結果は次のとおり。

| フェーズ         | 公式動作                     | ローカル再現結果                            |
| ---------------- | ---------------------------- | ------------------------------------------- |
| 再生開始         | プレイヤー生成後に再生       | `play` 1回                                  |
| チャプター移動   | 指定秒へシーク               | `currentTime` の書き込み1回、再生・停止なし |
| 次話切り替え     | 破棄完了後に再初期化して再生 | `destroy` → `initialize` → `play`           |
| ラストチャプター | 終端処理で停止               | `pause` 1回、`play` 0回、再生位置を維持     |

公式実装上の重要な不変条件は「ラストチャプターでは再初期化も再生もせず、停止位置を維持する」である。

## 7.6.2の再現結果

修正前の `VideoSwitchHandler` をフェイク動画、フェイクレンダラー、フェイクAPIで実行した。外部 `fetch` は例外に置き換えている。

### コメント表示までの構造的な遅延

切り替え処理は次の順で直列実行される。

1. `fetchVideoApiData()` が `fetchApiData()` を呼ぶ。
2. `populateComments()` が `fetchAllData()` を呼ぶ。
3. `fetchAllData()` が同じ動画に対して再び `fetchApiData()` を呼ぶ。
4. コメント取得完了後に全コメントをレンダラーへ追加する。

ローカル再現でも、1回の切り替えに対して `fetchApiData` 2回、`fetchAllData` 1回になった。したがって、少なくともメタデータ要求1往復分が不要な待ち時間として必ず加算される。初回読み込みも、自動検索結果の保存用とコメント取得用で別の `NicoApiFetcher` を作り、メタデータを2回取得している。

実ネットワークの所要時間は外部接続を遮断する方針上この調査では測らない。確認できたのは実時間ではなく、直列依存と重複要求による遅延構造である。

### 不要な初期化

エピソード変更処理はコントローラー側ですでに新しいレンダラーを生成・初期化した後、`resetVideoSource()` と `onVideoSwitch()` を続けて呼ぶ。`lastVideoSource` が `null` へ戻るため、同じ動画要素を再び「ソース変更」と判定する。

その結果、後段だけでも次が発生した。

- `clearComments`: 2回
- `destroy`: 1回
- `initialize`: 1回

コントローラー側の事前破棄・初期化を含めると、1回のエピソード変更でレンダラーの破棄と初期化がそれぞれ2回になる経路がある。再生開始時については、標準的な初期化経路だけならレンダラー初期化は1回であり、複数回になる条件はこのオフライン再現ではまだ確定していない。

### ラストチャプターの誤動作

公式の終端処理は `pause` のみで、再生位置を末尾に維持する。一方、現行 `VideoSwitchHandler` が終端を動画切り替えとして処理すると、ローカル再現では次の差が生じた。

- 停止状態自体は維持する。
- `currentTime` を末尾から `0` へ書き換える。
- コメントを2回クリアする。
- レンダラーを破棄・再初期化する。
- メタデータを2回取得する。

ユーザースクリプト内に動画の `play()` を直接呼ぶ箇所はないため、背景再生の直接原因を `VideoSwitchHandler` 単独とは断定できない。ただし、終端を切り替えとして誤認して再生位置とライフサイクルを変更することは、公式の終端不変条件への明確な違反である。公式側では本来の次話切り替え経路だけが `destroy` → `initialize` → `play` を行うため、終端イベントやソース変更判定がこの経路と競合すると背景再生へつながる余地がある。

## 7.6.3で反映した修正

1. `ended` イベントを動画切り替えの開始シグナルにしない。
2. `VideoSwitchHandler` にも終了済み動画を処理しない防御を置く。
3. コメント処理から動画の `currentTime` を変更せず、動画位置を公式プレイヤーだけに所有させる。
4. 検索結果に含まれるメタデータを保存へ利用し、コメント取得前の同一メタデータ要求を廃止する。
5. 切り替え処理では取得済みAPIデータから直接コメントを取得し、事前読み込み済みAPIデータも再利用する。

複数回のレンダラー初期化は今回の修正対象外とした。
